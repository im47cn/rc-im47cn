架构师Agent给出的第一轮答复：

```
技术栈选型提案
- 核心语言：Go 1.22+（推荐，天然的高并发处理能力，极低的资源占用，适合写轻量级通知 Worker）或 Python 3.11（如果追求极速交付）。暂定以 Go 为基准进行设计。
- 中间件/队列：Redis (Streams / List) 作为轻量级消息队列。不直接上 RabbitMQ/Kafka 是为了避免过度工程化（Over-engineering），Redis 的 ACK 机制足以满足现阶段的可靠性要求。
- 持久化层：SQLite 或 PostgreSQL，仅用于记录投递失败最终进入死信队列（DLQ）的通知日志，正常投递成功的“阅后即焚”，不占用数据库 I/O。

系统边界与核心决策提案
- 投递语义：确立 At-Least-Once（至少一次） 投递语义。业务系统不需要关心返回值，但我们需要确保请求被送达，因此必须引入重试机制（指数退避）与 ACK 确认机制。
- 系统边界（不解决的问题）：
  - 不解决下游接口的幂等性问题。由于我们是“至少一次”投递，下游必须自行保证幂等，我们在设计文档中显式声明此边界。
  - 不解决复杂的流控（Rate Limiting）隔离。第一版仅做全局并发控制，不针对单一供应商做精细化租户隔离。

待解决的关键技术问题
- 格式适配的复杂度（核心瓶颈）：由于不同供应商的 Header/Body 格式不同，你希望这个系统如何处理这种异构性？
  - 选项 A（动态模板）：系统支持配置 JSON 模板（如使用 Go template 或 JSON-to-JSON 映射规则），业务系统只发标准格式，通知系统通过配置动态渲染出供应商需要的格式。
  - 选项 B（硬编码适配器/打散透传）：系统不做任何解析，业务系统在发送通知时，自行组装好目标供应商所需的完整 URL、Header 和 Body（作为 Payload），通知系统仅作为一个纯粹的、盲目的（Blind）异步 HTTP 转发代理。
  - Kiro 的架构建议：作为第一版（MVP），强烈推荐选项 B。这符合减法思维，将格式拼装的复杂性留在业务端，通知系统得以保持极度的纯粹与高性能，完美避免过度设计。
- 极端失败策略：当外部供应商系统长期不可用（例如宕机 24 小时），重试达到最大次数后，你倾向于如何处理这些积压的通知？
  - 投递到死信队列（DLQ）并触发告警，等待人工干预或上游业务重发？
```

2. 第2轮沟通

```
问题 1：参数映射与函数预处理（加密）的实现路线
在 Java 动态处理 JSON 模板和参数映射（常量、变量、加密函数）时，为了保持 KISS（崇尚简洁）原则，我们有两条路线，你更倾向于哪一种？
- 路线 A（表达式语言驱动 - 强烈推荐）：利用 Spring 自带的 SpEL (Spring Expression Language) 或 AviatorEvaluator。
    实现方式：JSON 模板中直接写表达式，例如 {"sign": "#crypto.md5(#var.userId + #const.salt)", "version": "1.0"}。中台只需在上下文中注册 crypto 工具类和变量，由 SpEL 自动解析。
优势：代码量极少（几行代码即可搞定动态解析与函数调用），扩展性极强。
- 路线 B（结构化配置+策略模式）：在数据库中定义一套结构化的映射规则表（如：目标字段名、数据来源类型[变量/常量]、前置处理函数[MD5/AES]）。
    实现方式：代码里读取规则，通过 Switch-Case 或策略模式手动组装 JSON。
    优势：界面可视化配置非常友好，数据结构死板但可控，完全不担心安全沙箱问题。

问题 2：基于 Redis 的“供应商定制避让（延迟）”队列设计
由于不同供应商的重试时间间隔是定制的（比如：A 供应商 5s, 10s, 30s；B 供应商 1m, 5m, 15m），我们在开发环境使用 Redis，你希望如何实现这个精准的动态延迟重试？
- 方案 A（Redisson 延迟队列）：直接使用 Java 客户端 Redisson 提供的 RDelayedQueue（底层基于 Redis ZSet 和时间轮触发）。
    优势：开箱即用，代码极其优雅，非常符合 Java 开发者的习惯，完美契合 MVP。
- 方案 B（自研 ZSet 定时轮询）：将下一次需要重试的时间戳作为 Score 存入 Redis ZSet，系统启动一个单线程任务（或使用 @Scheduled）每秒 zrangebyscore 扫描到期数据。
    优势：不依赖 Redisson 相对复杂的分布式锁和底层 Lua 脚本，控制权完全在自己手里。
```

我的技术决策：

```
路线A：减法思维。该系统的配置大概率是技术人员来做，直接编写表达式不仅极大地降低了前端和数据库表结构设计的复杂度，还赋予了中台极强的动态扩展能力。
方案A：遵循 KISS 原则和 MVP 理念。开箱即用，用最小的底层组件开销换取了最可靠的“供应商级别定制避让”能力。

供应商A发生严重网络超时，极易导致线程被外部IO阻塞而耗尽。因此，动态为每个供应商创建独立的 Redisson 队列（如 delay_queue:supplier_A, delay_queue:supplier_B），并为每个队列配置独立的消费监听器。从源头上进行队列与消费线程的物理隔离。

5. 审查一致性并解决问题

```
问题1：
当前方案中过度依赖Redis中间件的稳定性，考虑到不同业务对通知成功的降级有不同的要求，引入“多模态降级策略矩阵”。
- 核心权益链路（如支付、下发订单）：数据绝对不能丢。如果中台由于 Redis 挂了而无法保证 At-Least-Once，宁可立刻向外报 503 阻断，让支付上游通过自己的本地事务或重试机制卡住流程，也绝不能盲目接收然后丢弃。
- 营销引流链路（如注册送积分、广告触达）：用户体验第一。外部广告供应商的接口挂了或者中台 Redis 挂了，绝对不能卡死用户的注册主流程，允许“尽力而为（Best-Effort）”，甚至允许极低概率的数据丢失。

问题2：
Spring 官方提供了一个针对只读/读写数据绑定的 SimpleEvaluationContext 且禁用了 T() 和构造函数，但是业务对数据格式转换（防腐层）的诉求不仅是“读写数据”，还要执行特定工具方法。SimpleEvaluationContext 内部的 MethodResolver 极为苛刻，它默认完全禁止在自定义变量上直接调用任何成员方法。如果我们采用它，上游模板写成 #crypto.md5(...) 就会直接报错。虽然可以通过 registerFunction 注册全局静态函数，但那会导致上游表达式必须写成标准非面向对象的 #md5(...)，失去了未来利用领域模型对象（比如 SupplierPolicy.isSatisfied()）进行精细化多态映射的扩展空间。

。。。。
