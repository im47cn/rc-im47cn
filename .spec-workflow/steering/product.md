# Product Overview

## Product Purpose

本产品为企业级 **API 通知中台（防腐层系统）**。旨在解决企业内部多个核心业务系统（如注册引流、付款订阅、商品购买等）与外部异构供应商（如三方广告系统、外部CRM、库存系统等）进行 HTTP(S) 异步通知时的强耦合、格式异构、网络不稳定导致的投递失败等问题。
系统作为一个高内聚的防腐层（Anti-Corruption Layer, ACL），将外部供应商 API 的复杂性、异构性、加解密规则和不稳定性完全屏蔽在中台内部，为上游业务系统提供统一、纯粹、高可用的标准通知投递服务。

## Target Users

* **内部业务线开发人员**：仅需对接中台的标准通知接口，无需关心外部供应商的 API 细节。
* **技术运营/系统管理员**：负责在通知中台通过 JSON 模板与 JSONata 表达式配置异构供应商的映射规则、避让策略与成功判定规则，并处理死信队列告警。

## Key Features

1. **高内聚防腐层（ACL）引擎**：支持通过 JSON 模板配置目标地址、Header、Path、Param、Body，支持基于 JSONata 的变量/常量映射、动态加密预处理函数。
2. **供应商级定制避让重试**：确立 At-Least-Once（至少一次）投递语义，支持为不同供应商独立配置定制化的指数避让策略（最大次数、初始延迟、时间间隔等）。
3. **多模态投递成功判定规则**：支持通过配置动态判定成功状态，可匹配特定的 HTTP 状态码（如 200）或解析响应报文内容（如严格等于/包含 "SUCCESS"，支持大小写敏感/不敏感配置）。
4. **死信队列（DLQ）降级机制**：重试达到最大次数后，通知自动转入死信队列并持久化至 MySQL，同时抛出标准化日志触发外部告警系统。

## Business Objectives

* **业务解耦**：实现内部核心业务系统与外部供应商系统的彻底解耦，外部 API 的变更升级不影响内部核心链路。
* **提高投递可靠性**：通过防腐层中台的自愈与异步重试机制，确保关键通知事件“至少送达一次”，规避因网络抖动或供应商短暂宕机导致的数据不一致。
* **降低研发成本**：新供应商接入由“开发代码适配”转变为“技术人员配置表达式”，将接入耗时从天级缩短至分钟级。

## Success Metrics

* **最终投递成功率**：排除 DLQ 降级后的首次投递成功率 ≥ 95%，含重试最终成功率 ≥ 99.9%。
* **配置上线效率**：技术人员通过配置完成一个异构供应商 API 的接入和测试，耗时不超过 **10 分钟**。
* **开发环境资源占用**：在 Redis+MySQL 开发环境下，中台常驻内存及 CPU 消耗极低，满足轻量级 MVP 交付标准。

## Product Principles

1. **减法思维（KISS）**：不追求过度工程化。在配置端依靠技术人员编写 JSONata 表达式，避免引入复杂的低代码可视化界面；在中间件端采用 Redisson 延迟队列，避免在 MVP 阶段引入过重的外部消息队列。
2. **上游极简原则**：复杂性留在中台内部，上游业务系统发送的 Payload 必须保持绝对的标准结构化元数据，不承担任何供应商适配逻辑。
3. **职责分离（SoC）**：通知中台仅专注于通知的高效、可靠投递与格式转换，不负责精细化的全局流控和监控看板展示，相关职责交由基础设施及外部日志告警系统承担。

## Monitoring & Visibility (if applicable)

* **系统级可观测性**：系统通过统一的日志格式输出投递指标。当单一外部供应商出现连续 3 次失败，且在特定窗口内失败占比超过 20% 时，系统打印特定级别的 Error 日志，由外部专业日志告警系统（如 ELK / Grafana Loki）进行拦截并触发告警。
* **死信可追溯性**：所有进入死信队列（DLQ）的记录完整保留原始 Payload、转换后的请求报文、历次失败原因以及最终触发时间，存储于 MySQL 中供技术人员直接检索和人工干预。

## Future Vision

* **分布式架构演进**：当企业业务规模扩大、QPS 达到数万级别时，系统能够无缝将底层的 Redis 延迟队列替换为生产级的 RocketMQ（使用定时消息/事务消息机制），持久化层从 MySQL 演进为分布式数据库，核心架构设计和 JSONata 引擎保持全兼容。
